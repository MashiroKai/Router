Token 计算原理 (Anthropic 格式)

Input（输入 token）

你发送给模型的全部内容都算 input token，包括：

  • System prompt — 系统指令、工具定义、项目上下文（Anthropic 顶层 system 字段）
  • 历史对话 — 之前的 user/assistant/tool 消息（每轮请求都会重新发送完整历史）
  • 当前用户输入 — 你这次实际输入的内容

API 返回 input_tokens = 总输入数，其中 cache_read_input_tokens 被缓存读取，cache_creation_input_tokens 被缓存写入。
显示的 Input = input_tokens − cache_read − cache_write（需要实际传输的部分）

Cache Read / Cache Write（缓存 token）

API 提供商（智谱 Anthropic 兼容端点）支持 Prompt Cache：

  • Cache Read — 命中已有缓存的 token，无需重新计算，成本更低
  • Cache Write — 首次写入缓存的 token，在后续请求中可被命中

点击 Cache Read 标签 → 高亮 system 和 assistant 历史消息（这些是缓存命中的主要来源）

Output（输出 token）

模型返回的全部 token，API 返回 output_tokens。包括文本输出和 tool_use 调用。

Thinking（思考）内容在 content blocks 中以 thinking 类型返回，不是单独的 reasoning token 计数。

Thinking（思考内容）

启用思考模式的模型会先进行内部推理，再给出回答。思考过程以 thinking content block 形式返回。

点击 OUTPUT 面板中的 Thinking 区域可展开查看完整思考内容。

数据来源

所有数字来自 API 响应的 usage 字段，不是客户端计算的。token 计数由服务端在处理请求时统计。

Context Window（上下文窗口）

每个模型有一个固定的上下文窗口大小（如 GLM-5.1 = 204,800 tokens）。这是单次请求能处理的最大 token 数量（输入 + 输出之和的上限）。

进度条显示的是 input_tokens / context_window，即输入占上下文窗口的比例。

  • 绿色 — 使用量 < 70%，安全
  • 黄色 — 使用量 70%–90%，注意
  • 红色 — 使用量 > 90%，接近上限

当上下文接近满时，模型可能截断输入或拒绝请求。长对话中 messages 不断累积是主要原因。
